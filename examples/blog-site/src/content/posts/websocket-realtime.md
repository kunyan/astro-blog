## 为什么需要 WebSocket

传统的 HTTP 协议是请求-响应模式，服务器无法主动向客户端推送数据。早期的实时方案如轮询（Polling）会产生大量无效请求，长轮询（Long Polling）虽然减少了请求次数，但每次仍需重新建立连接。WebSocket 协议在单个 TCP 连接上实现了全双工通信，服务端和客户端可以随时互相发送消息，是实时应用的理想选择。

## 协议握手过程

WebSocket 连接通过 HTTP Upgrade 机制建立。客户端发送包含 `Upgrade: websocket` 头的 HTTP 请求，服务器返回 `101 Switching Protocols` 确认协议升级。握手完成后，连接从 HTTP 切换为 WebSocket 协议，后续通信不再遵循 HTTP 格式，而是使用轻量级的帧（Frame）结构传输数据。

## 典型应用场景

### 即时通讯

聊天应用是 WebSocket 最经典的场景。用户发送消息后，服务器即时推送给目标用户或群组成员，延迟可以控制在毫秒级别。

### 实时数据推送

股票行情、体育比分、监控仪表盘等场景需要数据实时更新。WebSocket 允许服务器在数据变化时立即推送，比轮询方式更高效且时效性更强。

### 协同编辑

多人协同编辑文档时，每个用户的操作需要实时同步给其他参与者。WebSocket 的双向通信能力配合操作变换（OT）或 CRDT 算法，可以实现流畅的协同体验。

## 工程化实践

### 心跳保活

网络中间设备（如代理、防火墙）可能会关闭长时间空闲的连接。通过定时发送 Ping/Pong 帧保持连接活跃。建议心跳间隔设置为 30 秒，超过两次心跳未收到响应则判定连接断开。

### 断线重连

网络不稳定时连接可能意外断开。客户端应实现自动重连机制，采用指数退避策略避免重连风暴：首次立即重连，之后依次等待 1 秒、2 秒、4 秒……直到最大间隔（如 30 秒）。重连成功后需要恢复订阅状态和同步离线期间的消息。

### 消息序列化

WebSocket 支持文本和二进制两种消息类型。JSON 格式可读性好但体积较大，Protocol Buffers 或 MessagePack 等二进制格式在高频通信场景下更高效。建议统一消息格式，定义明确的消息类型和负载结构。

## 与 SSE 的对比

如果只需要服务端向客户端单向推送数据，Server-Sent Events（SSE）是更轻量的选择。SSE 基于 HTTP，天然支持断线重连和事件 ID 追踪，且不需要额外的协议升级。只有在需要双向通信时才应选择 WebSocket。
