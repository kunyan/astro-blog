## CI/CD 的核心价值

持续集成（CI）和持续部署（CD）的根本目标是缩短代码从编写到上线的反馈周期。开发者每次提交代码后，流水线自动执行构建、测试、部署等一系列步骤，在几分钟内给出明确的反馈。这种快速反馈机制让问题在刚引入时就被发现，修复成本远低于在发布前才集中测试。

## 流水线设计

### 阶段划分

一个典型的 CI/CD 流水线包含以下阶段：

1. **代码检查**：运行 linter 和格式化工具，确保代码风格一致
2. **单元测试**：执行速度快、覆盖面广的单元测试
3. **构建**：编译代码、打包产物、构建容器镜像
4. **集成测试**：验证组件间的协作是否正确
5. **部署到预发布环境**：在类生产环境中进行验证
6. **部署到生产环境**：经审批或自动推进到线上

阶段间应设置质量门禁（Quality Gate），前一阶段失败则中断流水线，避免浪费后续阶段的计算资源。

### 构建速度优化

流水线的速度直接影响开发体验。利用依赖缓存加速安装过程——缓存 `node_modules`、Go Module 缓存或 Docker 层缓存。将无依赖关系的任务并行执行，例如 lint 和单元测试可以同时运行。使用增量构建只处理变更的部分，而非每次全量构建。

## 部署策略

### 蓝绿部署

同时维护两套完全相同的生产环境。新版本部署到非活跃环境，验证通过后切换流量。回滚只需将流量切回旧环境，速度极快。缺点是需要双倍的基础设施资源。

### 金丝雀发布

先将新版本部署到少量实例（如 5% 的流量），观察错误率和性能指标。指标正常则逐步扩大流量比例，异常则立即回滚。这种渐进式发布策略将风险控制在最小范围。

### 特性开关

通过配置控制功能的开启和关闭，将代码部署和功能发布解耦。开发者可以随时合并代码到主干，但新功能只在开关打开时对用户可见。这简化了分支管理，也支持 A/B 测试和灰度发布。

## 基础设施即代码

流水线配置本身也应该版本化管理。使用 GitHub Actions、GitLab CI 或 Jenkins Pipeline 的 YAML/Groovy 文件定义流水线，与业务代码一起存放在代码仓库中。基础设施的变更同样需要经过代码审查和测试验证。

## 监控与回滚

部署不是终点。上线后应持续监控关键指标——错误率、响应时间、资源使用。设置自动告警阈值，异常时自动触发回滚或通知值班人员。保留最近几个版本的部署产物，确保在任何时候都能快速回退到已知的稳定版本。
